- **Tab 2 — Competitors / Outliers (LIVE, Phase 2):**
  - **What it does:** scans a list of competitor YouTube channels and surfaces **"outlier"** videos (over-performers) as topic ideas. Each outlier has a **"Make script"** button that hands its title to the Script tab — the loop: outlier → Make script → Script tab generates a script on that title.
  - **Outlier metric (the client's spec — views vs THAT channel's USUAL views, NOT raw views):** for each channel we sample its recent uploads (default 30, newest-first via the uploads playlist), compute the channel's **median** recent views, and flag any video whose `views >= median × OUTLIER_THRESHOLD` (default **3×**; `>=` is inclusive at the boundary). The reported **"multiple"** is `views / median` rounded to 1 dp (e.g. `3.4×`). Guards: a channel with `< 5` sampled uploads or a `median <= 0` yields **no** outliers (noisy/meaningless baseline); all-equal views → median == each view → no outliers. **Age-aware ranking:** `rankScore(multiple, ageDays)` adds a small boost (up to +50% to the **sort score only**, linear-decaying across a 14-day window) so a brand-new over-performer ranks above an equal-multiple older one — but age **never creates** an outlier (the `multiple` must already clear the threshold). The threshold/sample/cap are tunable constants at the top of `lib/youtube/data-api.ts` (`OUTLIER_THRESHOLD`, `RECENT_UPLOADS_SAMPLE`, `MIN_UPLOADS_FOR_MEDIAN`, `MAX_OUTLIERS`).
  - **Scan-on-open scheduling (the app runs LOCALLY, not 24/7):** target cadence is **every 2 days**. On opening the Competitors tab the page loads the saved state (`GET /api/competitors/scan`) and the YouTube key status (`GET /api/settings`); if a key is set **and** `lastScanAt` is `null` or **≥2 days** ago, it **auto-triggers one scan** (guarded to fire at most once per mount). A manual **"Scan now"** button is always available. The UI shows "last scanned X ago", a "next due in …" hint, and the cadence. The last-scan timestamp + results are persisted locally so they survive reloads.
  - **YouTube Data API v3 calls (quota-efficient — daily quota is 10,000 units):** per channel = **(1) `channels.list` `forHandle`** (1 unit) → channelId + uploads-playlist id; **(2) `playlistItems.list`** on that uploads playlist (1 unit/page of 50, newest-first — one page covers the 30-sample) → recent video ids; **(3) `videos.list`** `part=snippet,statistics` batched 50/call (1 unit/batch) → `viewCount` + `publishedAt` + `title`. That's **~3 units/channel**, so all 12 seed channels ≈ **36 units** — a tiny slice of the quota. We **deliberately avoid `search.list`** (100 units/call). A handle that can't be resolved (typo/deleted) yields `[]` and is recorded in a per-channel `errors[]` rather than aborting the whole scan; `scanAll` aggregates across channels, ranks by the age-aware score, and **caps to top 40** (logs when truncating).

## Performance tuning (2026-06-24) — two quality-safe changes

A real live run produced an **excellent ~2155-word script**, but surfaced two inefficiencies. Both fixes below were chosen because they are **quality-safe** (the first proven so in that live run; the second only trims verbosity and leaves all research/grounding untouched). Test count is now **185** (`npm test`, `npx tsc --noEmit`, `npm run lint`, `npm run build` all pass) — **+4** for these two changes.

1. **Draft (Stage 2): extended thinking turned OFF.** In the live run the draft used `thinking: { type: "adaptive" }` and the model spent its **entire** token budget thinking, emitting **0 words** of script. Re-running the same draft with thinking OFF produced the full, high-quality ~2155-word script in ~82s. Fix: `streamAnthropic()` no longer sends `thinking` and `max_tokens` is now **8000** (down from 64000 — a ~2000-word script is ~3000 tokens, so 8000 is a safe ceiling). `lib/models/anthropic.ts`. **Quality-safe: proven in the live test** — draft quality was excellent without thinking. (`researchAnthropic` Stage 1 and `converseAnthropic` fact-audit are **untouched**: they keep adaptive thinking + web search exactly as-is.) New assertions in `tests/anthropic-helpers.test.ts` confirm the draft sends no `thinking` and caps at 8000; `tests/engine-integration.test.ts` updated to match.

2. **Stage-1 outline: compactness nudge (prompt-level only).** The live run's Stage 1 produced a bloated **~39,000-character (~6000-word)** outline when the prompt asks for a **compact** blueprint. Fix: `buildResearchUser()` now appends a firm one-line instruction ("Keep the blueprint COMPACT: bullets only… This is a planning skeleton, not the script."). `lib/script/pipeline.ts`. **Quality-safe: output-only** — the web search tool, the search count (`max_uses`), the research thinking, and `researchAnthropic`'s `max_tokens` are all **unchanged**, so grounding/research depth is identical; only the *verbosity* of the written outline is reduced. `tests/pipeline.test.ts` updated to expect the appended line (and the structural-frame tests now check the closer is present + the nudge is the final line).

### Fixed in source (with regression tests)

- **`buildDraftSystem` $-injection corruption (should-fix → fixed).** The Stage-1 blueprint is LLM free text and can contain `$` sequences (`$&`, `` $` ``, `$'`, `$$`, `$1`). It was being passed as the *string* second arg to `String.prototype.replace`, which interprets those as replacement patterns — silently corrupting the system prompt (e.g. `$&` re-inserted the matched `<story_structure>` tag block + the `{insert blueprint}` placeholder; `` $` ``/`$'` garbled the prompt and left `<story_length>` unsubstituted). Fixed by passing the replacement as a **function** (`() => structure`), which inserts it verbatim. `lib/script/pipeline.ts`; regression tests in `tests/pipeline.test.ts` ("$-injection from the blueprint") — these FAIL against the old code.
- **Fact-audit thinking config: `budget_tokens` → adaptive (should-fix → fixed).** `converseAnthropic` used `thinking: { type: "enabled", budget_tokens: 8000 }` on `claude-sonnet-4-6`. That config is **deprecated** on 4.6 (still functional, but adaptive is the GA path per the claude-api guidance) and carried a latent `budget_tokens < max_tokens` foot-gun. Switched to `thinking: { type: "adaptive" }`, matching Stage 1/2 of the pipeline. `lib/models/anthropic.ts`; assertion updated in `tests/fact-audit-stream.test.ts`.

### Verified correct (non-issues — confirmed against the installed `@anthropic-ai/sdk@0.105.0` + Anthropic docs)

- **Web search tool shape:** `{ type: "web_search_20250305", name: "web_search", max_uses }` is a valid `ToolUnion` member in the installed SDK and is the correct GA shape for `messages.stream` on Sonnet 4.6. ✅
- **Multi-turn + thinking with plain-text prior turns (the suspected blocker):** the fact audit replays prior assistant turns as plain text (no thinking/tool_use blocks) with thinking enabled. **This is accepted (HTTP 200), NOT a 400.** The Anthropic extended-thinking docs state: *"you can omit `thinking` blocks from prior assistant role turns."* A 400 only fires when thinking blocks are (a) **modified**, or (b) the **latest** assistant turn carries a `tool_use` without its thinking block *during a tool loop*. Neither applies: each phase is a fresh request whose final turn is a `user` confirmation, and web search resolves fully server-side inside one `.finalMessage()` — there's no client-replayed `tool_use` turn. Design is sound. (See the long comment on `converseAnthropic`.)
- **`extractText` / `extractSources` vs the real Sonnet-4.6 content shape:** both correctly skip `thinking` and `server_tool_use` blocks and read URLs only from `web_search_tool_result` blocks; `extractSources` guards the error variant (`content` is an object, not an array) via `Array.isArray`. Covered by new tests in `tests/anthropic-helpers.test.ts` using realistic thinking → server_tool_use → web_search_tool_result → text sequences. ✅
- **No-prefill rule:** every messages array ends on a `user` turn (Stage 1, Stage 2, and all 3 audit phases). ✅
- **NDJSON framing vs script text (risk: a `}`-heavy / newline-laden / protocol-looking token):** `encodeEvent` uses `JSON.stringify`, which escapes embedded newlines, so each event is always exactly one physical line — token text that contains real newlines, braces, or even a literal `{"type":"done"}` string round-trips as token text, never re-parsed as a second event. Covered by new tests in `tests/protocol.test.ts`. ✅

### Input robustness (operator-typed title — low severity, documented not fixed)

The title flows into `buildResearchUser` raw. A title containing the literal delimiter `</Input>` (or `<story_structure>`, "ignore previous instructions", newlines, backticks) places injected text *outside* the first `<Input>` block but **keeps the wrapper structurally well-formed** (always opens `<Input>` / closes `</Input>`, metadata lines always present — covered by new "odd & hostile titles" tests). This is **not** a privilege-boundary issue: the title is typed by the operator (Marcus), running his own script with his own key — there is no untrusted third party. Hardening (escaping/stripping the delimiter) is cheap if desired but was left as-is since there's no real injection boundary to defend.
